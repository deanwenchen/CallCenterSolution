# CallCenter AI — 代码解析

> 本文档帮助你读懂代码结构、读懂流程、读懂框架设计意图。适合第一次接触项目的人。

---

## 一、先看全貌：这个项目到底在做什么

一句话总结：**用户说"我要退款"，系统自动识别意图、启动流程、追问缺失参数、完成业务操作，全程不需要人管。**

实现方式：

1. 用户输入一行文字
2. LLM 判断意图（退款 / 换货 / 闲聊）
3. 如果是退款 → 启动一个由 7 个步骤组成的工作流
4. 工作流中间会暂停，等用户确认后再继续
5. 最终调用财务服务完成退款

整个系统围绕 **Microsoft Agent Framework (MAF)** 这个 SDK 构建，它提供了三个核心能力：
- `AIAgent` — 让 LLM 像"人"一样思考
- `Workflow` — 把业务步骤串成有向图
- `RequestPort` — 让工作流暂停下来问人要答案

---

## 二、项目结构速览

```
src/
├── CallCenter.Shared/          ← 最底层：接口定义 + Mock 数据
├── CallCenter.Framework/       ← 中间层：横切关注点（安全、审计、事件总线、Saga...）
├── CallCenter.Workflows/       ← 业务层：退款流程 + 换货骨架
├── CallCenter.AgentHost/       ← 编排层：入口路由 + 技能注册
└── CallCenter.ConsoleDemo/     ← 入口层：命令行演示程序
```

**依赖方向**：ConsoleDemo → AgentHost → Workflows → Framework → Shared

每层做什么、不做什么，分得很清楚：

| 层 | 做什么 | 不做什么 |
|---|--------|----------|
| Shared | 定义接口（IOrderMcpClient）和数据模型（OrderInfo） | 不包含实现逻辑 |
| Framework | 提供横切能力（安全、审计、事件总线、Saga） | 不包含业务规则 |
| Workflows | 实现具体业务流程（退款、换货） | 不关心安全、审计怎么实现 |
| AgentHost | 做意图识别、路由决策、技能管理 | 不包含工作流逻辑 |
| ConsoleDemo | 把一切组装起来演示 | 不包含业务逻辑 |

## 三、逐层解析

### 3.1 Shared 层：接口 + 数据模型

**作用**：定义业务边界。告诉上层"你能调什么服务"。

```csharp
// 订单服务接口
public interface IOrderMcpClient
{
    Task<OrderInfo?> GetOrderAsync(string orderId);
    Task<List<OrderInfo>> GetRecentOrdersAsync(string userId);
}

// 财务服务接口
public interface IFinanceMcpClient
{
    Task<RefundResult> RefundAsync(string orderId, decimal amount);
}

// 会员服务接口
public interface IMemberMcpClient
{
    Task<CouponInfo?> GetCouponAsync(string userId);
    Task<bool> RestoreCouponAsync(string userId, string couponId);
}
```

**为什么叫 MCP**：MAF 的 Model Context Protocol 约定，外部系统都通过这个协议接入。当前使用 Mock 实现，未来换真实实现时，只需实现这三个接口。

**Mock 订单数据**：

```csharp
// 三个测试订单，覆盖不同退款场景
A001 — 蓝牙耳机 ¥299，3 天前，已签收，可退 ✅
A002 — 定制T恤 ¥159，30 天前，已签收，不可退（超期 + 定制）
A003 — 手机壳 ¥39，1 天前，已发货，不可退（未签收）
```

### 3.2 Framework 层：横切关注点

**作用**：提供"所有业务都能用的通用能力"。业务层只需要注入依赖，不需要自己实现。

#### 3.2.1 6 层聊天管道（StandardPipelineFactory）

**这是整个系统最精巧的设计之一。**

```
用户输入 → [6] SafetyInput → [5] Logging → [4] Compaction → [3] ToolApproval → [2] LLM → [1] SafetyOutput → 用户输出
```

每一层都是一个 `DelegatingChatClient`（包装器），层层套娃：

```csharp
// 洋葱模型：最外层最后返回，但最先执行
client = baseClient;                          // [2] 最内层：原始 LLM
client = new SafetyOutputDelegatingClient(client); // [1] 输出脱敏
client = new ToolApprovalDelegatingClient(...);    // [3] 工具审批
client = CreateCompactedClient(...);               // [4] 上下文压缩
client = new LoggingDelegatingClient(...);         // [5] 请求/响应日志
client = new SafetyInputDelegatingClient(client);  // [6] 最外层：输入过滤
```

**为什么叫洋葱模型**：请求从外层进入，逐层向内；响应从内层返回，逐层向外。

| 层 | 作用 | 代码位置 |
|---|------|----------|
| SafetyInput | PII 脱敏 + 关键词拦截 + 注入检测 | `Safety/SafetyInputFilter.cs` |
| Logging | 记录每次请求/响应到 .logs/ | `Logging/JsonlLogger.cs` |
| Compaction | 超过 8000 token 或 8 轮时压缩历史 | `Compaction/CompactionExtensions.cs` |
| ToolApproval | 审批工具调用（当前默认全部通过） | `ToolApproval/ToolApprovalDelegatingClient.cs` |
| LLM | 真实的 DashScope Qwen 调用 | — |
| SafetyOutput | AI 输出的 PII 脱敏 | `Safety/SafetyOutputFilter.cs` |

**安全输入管道的执行顺序**：

```
用户输入 → PII 脱敏 → 关键词检查 → 注入检测 → 通过则放行，否则抛 SafetyViolationException
```

#### 3.2.2 事件总线（InMemoryBusinessEventBus）

**作用**：把工作流和外部关注点（通知、分析、监控）解耦。

```csharp
// 工作流只做一件事：发布事件
await _eventBus.PublishAsync(new RefundCompletedEvent(...));

// 谁关心谁订阅，不需要修改工作流代码
eventBus.Subscribe<RefundCompletedEvent>(async e =>
{
    Console.WriteLine($"退款完成: 订单{e.OrderId}, 金额 {e.RefundAmount}");
});
```

**设计亮点**：
- `Subscribe` 返回 `IDisposable`，用 `using` 或 `.Dispose()` 可以随时取消订阅
- `PublishAsync` 采用 fire-and-forget，慢处理器不阻塞发布者
- 生产环境可替换为 RabbitMQ / Kafka 等持久化消息代理

#### 3.2.3 审计日志（AuditLogger）

**作用**：把工作流每一步的输入/输出写成带 SHA256 哈希链的日志，事后可以验证是否被篡改。

```
日志文件格式：.audit/{sessionId}.jsonl
每行一条 AuditLogEntry：
{
  "Timestamp": "2026-06-02T01:37:38...",
  "ExecutorId": "GetOrder",
  "EventType": "step_start",
  "InputJson": "{...}",
  "PreviousHash": "abc123...",    ← 指向上一步的哈希
  "Hash": "def456..."            ← 本行的哈希（用 PreviousHash 参与计算）
}
```

**哈希链原理**：每行的 `PreviousHash` 是上一行的 `Hash`。如果有人篡改了第 3 行，第 4 行的 `PreviousHash` 就对不上了。

```csharp
// 验证方法
var result = await auditLogger.VerifyChainAsync(sessionId);
// result.Status = Valid / Invalid / FileNotFound
```

#### 3.2.4 Saga 补偿（SagaBuilder）

**作用**：有副作用的业务步骤，失败了要回滚。

```csharp
var saga = new SagaBuilder()
    .OnFailure("ExecuteRefund", async ct => {
        // 补偿：恢复优惠券
        await RestoreCouponAsync();
    })
    .WithRetry(3, TimeSpan.FromSeconds(1), TimeSpan.FromSeconds(2));

await saga.ExecuteAsync(async ct => {
    // 主操作：执行退款
    await RefundAsync(orderId, amount, ct);
});
```

**执行逻辑**：
1. 先执行主操作，成功则返回
2. 失败则重试（最多 3 次，间隔递增）
3. 重试全部耗尽，依次执行所有补偿操作

#### 3.2.5 会话存储（InMemorySessionStore）

**作用**：在一次会话生命周期内保存各种状态。

```
存储内容：
- activeWorkflow → "RefundWorkflow"（当前在跑的工作流名）
- lastCheckpoint → { StepNumber: 3, ... }（断点信息，用于恢复）
- lastActivity → DateTime.UtcNow（超时检测用）
- pendingOrderId → "A001"（动态追问时临时存的订单号）
- order → OrderInfo 对象（GetOrderExecutor 查到的订单）
- refundAmount → decimal（CheckRefundRule 算出的退款金额）
```

**数据结构**：两层字典 `ConcurrentDictionary<scope, ConcurrentDictionary<key, value>>`

**为什么用 ConcurrentDictionary**：线程安全，多线程并发读写不会出问题。

#### 3.2.6 结构化输出解析器（StructuredOutputParser）

**作用**：LLM 返回的 JSON 文本 → 强类型 C# 对象。

```csharp
// LLM 返回的文本（可能带 Markdown 围栏）
var json = "```json\n{\"intent\": \"refund\", \"orderId\": \"A001\"}\n```";

// 解析为强类型对象
var parser = new StructuredOutputParser<IntentResult>();
var result = parser.Parse(json);
// result.Intent = "refund", result.OrderId = "A001"
```

**关键处理**：先剥离 ` ```json ... ``` ` 围栏，再 `JsonSerializer.Deserialize`。

### 3.3 Workflows 层：业务逻辑

**作用**：具体的业务流程定义。一个业务 = 一个 Workflow + 若干 Executor。

#### 3.3.1 工作流图是怎么构建的

```csharp
// RefundWorkflow.cs
public static Workflow Build(...)
{
    // 1. 创建两个端口（人工介入点）
    var infoPort = RequestPort.Create<RefundSignal, RefundIntent>("RefundInfoPort");
    var confirmPort = RequestPort.Create<ConfirmRefundRequest, UserConfirmation>("RefundConfirmPort");

    // 2. 创建 7 个 Executor（节点）
    var getOrder = new GetOrderExecutor(orderService);
    var checkRule = new CheckRefundRuleExecutor();
    var waitConfirm = new WaitUserConfirmExecutor();
    var doRefund = new ExecuteRefundExecutor(financeService);
    var restoreCoupon = new RestoreCouponExecutor(memberService);
    var notify = new SendNotificationExecutor(eventBus);
    var denied = new RefundDeniedExecutor();

    // 3. 用 builder 连线（有向边）
    var builder = new WorkflowBuilder(getOrder);  // GetOrder 是起点
    builder.ForwardMessage<OrderFound>(getOrder, checkRule);      // 查单成功 → 规则校验
    builder.ForwardMessage<RefundSignal>(getOrder, infoPort);     // 需要订单号 → 端口
    builder.AddEdge<RefundRuleResult>(checkRule, waitConfirm, r => r?.IsEligible == true);  // 合格 → 等待确认
    builder.AddEdge<RefundRuleResult>(checkRule, denied, r => r?.IsEligible == false);      // 不合格 → 拒绝
    builder.ForwardMessage<ConfirmRefundRequest>(waitConfirm, confirmPort);  // 确认请求 → 端口
    builder.AddEdge(confirmPort, doRefund);       // 用户确认 → 执行退款
    builder.ForwardMessage<RefundExecuted>(doRefund, restoreCoupon);  // 退款完成 → 恢复优惠券
    builder.ForwardMessage<CouponRestored>(restoreCoupon, notify);    // 优惠券恢复 → 通知

    return builder.WithOutputFrom(notify, denied).Build();  // 通知或拒绝都是最终输出
}
```

**关键理解**：
- `WorkflowBuilder` 是起点 → 后面用 `ForwardMessage` 和 `AddEdge` 连线
- `ForwardMessage<T>` — 无条件转发：收到 T 类型消息就转发给下一个节点
- `AddEdge<T>(..., condition)` — 条件转发：只有 condition 返回 true 才转发
- `RequestPort` — 特殊节点：收到消息后工作流暂停，等外部回复再继续
- `WithOutputFrom(notify, denied)` — 这两个节点的输出作为工作流的最终输出

#### 3.3.2 消息类型体系

所有消息都是 `record`，不可变，线程安全：

```
流程输入:    RefundIntent(OrderId?, UserId?)
             ↓
流程内部:    OrderFound → RefundRuleResult → ConfirmRefundRequest → UserConfirmation
             ↓
流程输出:    RefundExecuted → CouponRestored → RefundNotification(string Message)
```

#### 3.3.3 7 个 Executor 逐一解析

**GetOrderExecutor（第一步：查订单）**
```
输入: RefundIntent（可能带订单号，也可能不带）
输出: OrderFound（查到了）或 RefundSignal.NeedOrderId（需要用户提供）
逻辑:
  1. 如果没订单号 → 发 NeedOrderId 信号给端口
  2. 有订单号 → 调用 IOrderMcpClient.GetOrderAsync
  3. 订单不存在 → 也发 NeedOrderId
  4. 订单存在 → 保存到 state，发 OrderFound 给下一步
```

**CheckRefundRuleExecutor（第二步：规则校验）**
```
输入: OrderFound
输出: RefundRuleResult(IsEligible, Reason, RefundAmount)
规则:
  1. 超过 7 天 → 不合格
  2. 未签收 → 不合格
  3. 定制商品 → 不合格
  4. 合格 → 计算退款金额（有优惠券扣 20 元），保存到 state
```

**WaitUserConfirmExecutor（第三步：等待用户确认）**
```
输入: RefundRuleResult
输出: ConfirmRefundRequest（发给 ConfirmPort）
逻辑:
  把金额、订单号、商品名发给端口，工作流暂停
  用户在控制台看到信息，回复"确认"或"取消"
```

**ExecuteRefundExecutor（第四步：执行退款）**
```
输入: UserConfirmation(Confirmed: true/false)
输出: RefundExecuted(Result)
逻辑:
  Confirmed = false → 返回 RefundExecuted(null)（不执行退款）
  Confirmed = true  → 从 state 读订单和金额，调用财务服务退款
```

**RestoreCouponExecutor（第五步：恢复优惠券）**
```
输入: RefundExecuted
输出: CouponRestored(CouponId)
逻辑:
  Result = null → 跳过（用户取消了，不用恢复）
  Result != null → 调用会员服务恢复优惠券
```

**SendNotificationExecutor（第六步：通知用户）**
```
输入: CouponRestored
输出: RefundNotification(string Message)
逻辑:
  refundResult = null → 输出"退款流程已结束"
  refundResult != null → 发布 RefundCompletedEvent，输出退款完成消息
```

**RefundDeniedExecutor（替代路径：拒绝）**
```
输入: RefundRuleResult(IsEligible = false)
输出: RefundNotification("退款被拒绝: 原因")
逻辑: 直接把拒绝原因输出给用户，流程结束
```

### 3.4 AgentHost 层：入口路由

**作用**：承接用户输入，做意图识别，决定下一步怎么走。

```
用户输入 → EntryPoint.ProcessAsync()
                ↓
        检查超时（30min 警告 / 60min 终止）
                ↓
        检查是否有活跃工作流
                ↓
        ┌─ 有活跃工作流 ──→ 识别意图
        │                   ├─ 相同意图 → ResumeExisting（继续）
        │                   ├─ 不同意图 → IntentSwitch（终止旧的，启动新的）
        │                   └─ 闲聊 → NoIntent（回复但不终止流程）
        │
        └─ 无活跃工作流 ──→ 识别意图
                            ├─ refund → StartWorkflow（启动退款流程）
                            ├─ greeting → NoIntent（你好！）
                            └─ unknown → NoIntent（不太明白...）
```

**IntentResult 结构**（LLM 返回的 JSON 反序列化得到）：

```csharp
public record IntentResult(
    [JsonPropertyName("intent")] string Intent,      // "refund" / "greeting" / "unknown"
    [JsonPropertyName("workflow")] string? Workflow, // "RefundWorkflow"
    [JsonPropertyName("orderId")] string? OrderId    // "A001" 或 ""
);
```

### 3.5 ConsoleDemo 层：组装与演示

**作用**：把所有东西组装起来，提供一个可交互的命令行界面。

**组装顺序**（Program.cs 从上到下）：

```
1. 读取环境变量（API Key、模型名）
2. 创建 LLM 客户端
3. 创建 Mock 服务
4. 创建事件总线、会话存储、Checkpoint 管理器
5. 订阅退款完成事件
6. 构建工作流图
7. 注册 Agent 技能
8. 构建 6 层聊天管道
9. 创建审计日志记录器
10. 创建 EntryPoint
11. 创建 stdin 通道（解耦输入和事件循环）
12. 进入主循环：读用户输入 → ProcessAsync → 根据结果执行
```

**主循环流程**：

```
while (true) {
    用户输入 → entryPoint.ProcessAsync()
    
    switch (result) {
        case ResumeExistingResult → ResumeWorkflow(从断点续跑)
        case StartWorkflowResult  → RunWorkflow(启动新工作流)
        case TimeoutResult        → 显示超时信息
        case IntentSwitchResult   → 显示意图切换信息
        case NoIntentResult       → 显示闲聊回复
    }
}
```

**RunWorkflow 函数**：

```
执行工作流，监听事件流：
  - RequestInfoEvent → 通过 HandleRequest 转成用户交互（问订单号 / 问确认）
  - WorkflowOutputEvent → 显示最终结果，清理状态，返回
  - SuperStepCompletedEvent → 保存断点
  - WorkflowErrorEvent → 显示错误，尝试 Saga 补偿，清理状态，返回

特殊处理：如果工作流需要订单号，拿到后更新 currentMessage 并重新运行整个工作流
```

**HandleRequest 函数**：

```
根据 RequestPort 的类型，做不同的交互：
  - RefundSignal.NeedOrderId → "请提供订单号: " → 读取输入 → 返回 RefundIntent
  - ConfirmRefundRequest → 显示订单信息 → "确认？" → 读取输入
      - "确认" → 返回 UserConfirmation(true)
      - "取消" → 返回 UserConfirmation(false)
      - 其他 → 重新意图识别，判断是闲聊、新意图还是无法识别
```

## 四、以"我要退款"为例，完整走一遍代码

### 场景：用户输入"我要退款，订单A001"，然后回复"确认"

```
第 1 步: ConsoleDemo 主循环
  Console.ReadLine() → "我要退款，订单A001"
  entryPoint.ProcessAsync("demo-session", "我要退款，订单A001", refundWorkflow)

第 2 步: EntryPoint.ProcessAsync
  _sessionStore.SetAsync("lastActivity", DateTime.UtcNow)  ← 更新活动时间
  CheckTimeoutAsync → null（首次输入，不检查超时）
  GetActiveWorkflowAsync → null（没有活跃工作流）
  
  RecognizeIntentAsync("我要退款，订单A001"):
    _intentAgent.RunAsync(...) → LLM 返回
      {"intent": "refund", "workflow": "RefundWorkflow", "orderId": "A001"}
    parser.Parse(...) → IntentResult("refund", "RefundWorkflow", "A001")
  
  newIntent.Intent == "refund" → 
    SetActiveWorkflowAsync("RefundWorkflow")
    return StartWorkflowResult(RefundIntent("A001", "U100"))

第 3 步: ConsoleDemo 主循环 switch
  case StartWorkflowResult → RunWorkflow(refundWorkflowWithAudit, RefundIntent("A001", "U100"))

第 4 步: RunWorkflow → 启动工作流
  InProcessExecution.RunStreamingAsync(workflow, RefundIntent("A001", "U100"))
  监听事件流...

第 5 步: 工作流执行 — GetOrderExecutor
  输入: RefundIntent("A001", "U100")
  message.OrderId = "A001" → 有订单号
  _orderService.GetOrderAsync("A001") → OrderInfo(蓝牙耳机, ¥299, delivered)
  context.QueueStateUpdateAsync("order", order)  ← 保存到 state
  context.SendMessageAsync(OrderFound(order))    ← 发给下一步
  
  事件: SuperStepCompletedEvent → 保存断点

第 6 步: 工作流执行 — CheckRefundRuleExecutor
  输入: OrderFound(order)
  daysSinceOrder = 3 → ≤ 7 ✅
  order.Status = "delivered" ✅
  order.Category = "electronics" ≠ "custom" ✅
  refundAmount = 299 - 20(优惠券) = 279
  context.QueueStateUpdateAsync("refundAmount", 279)
  返回: RefundRuleResult(true, null, 279, "A001", "蓝牙耳机")
  
  条件边: IsEligible == true → 转发给 WaitUserConfirmExecutor

第 7 步: 工作流执行 — WaitUserConfirmExecutor
  输入: RefundRuleResult(true, null, 279, "A001", "蓝牙耳机")
  context.SendMessageAsync(ConfirmRefundRequest(279, "A001", "蓝牙耳机"))
  消息通过 ConfirmPort 发出 → RequestInfoEvent

第 8 步: ConsoleDemo 收到 RequestInfoEvent
  HandleRequest 被调用:
    request.TryGetDataAs<ConfirmRefundRequest> → true
    打印: "订单 A001: 蓝牙耳机 ¥279.00"
    打印: "确认退款？(回复'确认'或'取消'): "
    Console.ReadLine() → "确认"
    返回: ExternalResponse(UserConfirmation(true))
  run.SendResponseAsync(response) → 注入到工作流

第 9 步: 工作流执行 — ExecuteRefundExecutor
  输入: UserConfirmation(true)
  message.Confirmed == true → 执行退款
  从 state 读: order = OrderInfo(...), refundAmount = 279
  _financeService.RefundAsync("A001", 279) → RefundResult("RF-xxx", "success")
  context.QueueStateUpdateAsync("refundResult", result)
  返回: RefundExecuted(result)

第 10 步: 工作流执行 — RestoreCouponExecutor
  输入: RefundExecuted(result != null)
  _memberService.RestoreCouponAsync("U100", "CPN-2024") → true
  返回: CouponRestored("CPN-2024")

第 11 步: 工作流执行 — SendNotificationExecutor
  输入: CouponRestored("CPN-2024")
  从 state 读: refundResult = RefundResult(...)
  _eventBus.PublishAsync(RefundCompletedEvent(...))  ← 发布事件
  context.YieldOutputAsync("退款 RF-xxx 已处理完成，预计 3-5 个工作日到账")
  
  事件: WorkflowOutputEvent → ConsoleDemo 显示结果
  ConsoleDemo: sessionStore.RemoveAsync("activeWorkflow")  ← 清除活跃状态
  RunWorkflow return → 回到主循环
```

## 五、框架优势

| 优势 | 说明 | 代码体现 |
|------|------|----------|
| **业务与框架分离** | 退款规则只在 Workflows 层，Framework 只关心横切能力 | 各层职责清晰，互不干扰 |
| **工作流可暂停** | RequestPort 让流程在需要用户输入时暂停，收到回复后恢复 | `WaitUserConfirmExecutor` 发 ConfirmPort → 暂停 → 用户回复 → 继续 |
| **可扩展性强** | 新增业务只需：创建 Workflow → 编写 Executors → 注册 Skill | Exchange 骨架已按同样模式搭好，只需实现业务逻辑 |
| **消息类型驱动** | 工作流节点通过消息类型传递数据，天然类型安全 | `Executor<OrderFound, RefundRuleResult>` — 输入输出都是强类型 |
| **条件边路由** | 通过 lambda 表达式决定下一步走向，灵活且可读 | `AddEdge<RefundRuleResult>(..., r => r?.IsEligible == true)` |
| **6 层安全管道** | 所有 LLM 调用都经过安全过滤，业务代码不需要关心 | `StandardPipelineFactory.CreatePipeline(...)` |
| **审计可验证** | 每步操作带 SHA256 哈希链，可事后验证完整性 | `AuditLogger.VerifyChainAsync(sessionId)` |
| **失败补偿** | Saga 模式支持重试 + 回滚，避免半成功状态 | `SagaBuilder.OnFailure(...).WithRetry(...).ExecuteAsync(...)` |
| **事件解耦** | 工作流只发布事件，外部系统按需订阅 | `eventBus.PublishAsync(new RefundCompletedEvent(...))` |
| **LLM 意图识别** | 不依赖关键词匹配，更接近真实场景 | `ChatClientAgent` + `StructuredOutputParser<IntentResult>` |

## 六、框架劣势与改进建议

| 劣势 | 影响 | 改进建议 |
|------|------|----------|
| **无真实持久化** | 服务重启后所有会话状态丢失 | 实现 RedisSessionStore，持久化 activeWorkflow / checkpoint |
| **硬编码工作流名** | `activeWorkflow == "RefundWorkflow"` 判断是硬编码的 | 用 IntentResult.Workflow 动态匹配 |
| **意图识别只做一次** | 用户输入"我要退款"后，中间交互不再经过意图识别 | 可在 ConfirmPort 回复后自动做意图识别（IR-05 已部分实现） |
| **无 Web 接入** | 只有 ConsoleDemo，没有 HTTP/API 入口 | 实现 CallCenter.Gateway（PRD 已设计） |
| **工具审批默认通过** | `DefaultToolApprovalAgent` 始终返回 true | 实现基于规则的审批逻辑 |
| **KeywordFilter 不可配置** | 拦截词写死在代码里 | 改为从配置文件/数据库读取 |
| **无测试覆盖** | 没有单元测试，依赖手动验收 | 为 Executor 编写单元测试，Mock MCP 服务 |
| **Exchange 骨架无业务逻辑** | 所有 Executor 都 throw NotImplementedException | v2 实现具体业务规则 |
| **单会话限制** | ConsoleDemo 固定 sessionId = "demo-session" | Gateway 层支持多用户多会话 |
| **LLM 调用异常被吞** | `RecognizeIntentAsync` 的 catch 块返回 null，不记录日志 | 至少记录警告日志，方便排查 |

## 七、关键设计模式速查

| 模式 | 用在哪儿 | 作用 |
|------|---------|------|
| **洋葱模型（Decorator）** | StandardPipelineFactory 的 6 层委托链 | 每层包装 inner client，逐层增强 |
| **有向图（DAG）** | WorkflowBuilder + Executor + Edge | 用代码描述业务流程图 |
| **发布/订阅** | InMemoryBusinessEventBus | 解耦工作流和外部关注点 |
| **Saga** | SagaBuilder | 失败重试 + 补偿回滚 |
| **工厂方法** | StandardPipelineFactory、RefundWorkflow.Build | 封装复杂对象的创建 |
| **记录类型（record）** | 所有消息类型（IntentResult、OrderFound...） | 不可变数据，天然线程安全 |
| **通道（Channel）** | ConsoleDemo 的 inputChannel | 解耦控制台输入和事件循环 |
