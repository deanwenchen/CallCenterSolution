### Task 3: Intent re-recognition during confirmation prompts

<read_first>
- src/CallCenter.ConsoleDemo/Program.cs (HandleRequest method, lines 228-262, ConfirmRefundRequest block lines 244-259)
- src/CallCenter.AgentHost/EntryPoint.cs (RecognizeIntentAsync method, lines 58-74)
</read_first>

<action>
In `HandleRequest`, replace the confirmation prompt logic for non-"确认"/"取消" responses:

Current behavior (lines 254-257): unrecognized reply → treat as cancel

New behavior:
1. When user reply is neither "确认" nor "取消":
   a. Suspend the current workflow (do NOT send UserConfirmation)
   b. Re-recognize intent by calling `EntryPoint.RecognizeIntentAsync` with the user's reply
   c. If intent is "greeting" → return chitchat reply ("你好！有什么可以帮助你的？")
   d. If intent is "unknown" → return fallback reply ("抱歉，我不太明白...")
   e. If intent is a new workflow intent (e.g., "complaint") → set `suspendedWorkflow` in session store, clear `activeWorkflow`, print intent switch message
   f. Print the re-recognized intent response instead of processing confirmation

Implementation approach:
- `HandleRequest` cannot call async methods or access EntryPoint directly
- Change `HandleRequest` signature to accept `Func<string, Task<IntentResult?>> recognizeIntent` and `Func<string, string, Task> onIntentSwitch` callbacks
- In Program.cs main loop, pass `entryPoint.RecognizeIntentAsync` as the callback
- For unrecognized confirm replies: call recognizeIntent → return appropriate response
</action>

<acceptance_criteria>
- `HandleRequest` signature includes `Func<string, Task<IntentResult?>>` parameter for intent re-recognition
- Unrecognized confirm reply triggers `RecognizeIntentAsync` call
- "我要投诉" → intent recognized as non-refund → returns chitchat/fallback response
- "你好" → greeting intent → returns "你好！有什么可以帮助你的？"
- Source: `Program.cs` contains `RecognizeIntentAsync` call within HandleRequest or its callback
- Source: HandleRequest no longer has `reply == "取消"` as default fallback for unrecognized input
</acceptance_criteria>
